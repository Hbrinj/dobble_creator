# alpha-aware-packing

Compute each uploaded image's true silhouette geometry (alpha bounding box + smallest enclosing circle of opaque pixels) and feed that into the card render pipeline so visible art fills the slot circle instead of an oversized circumscribing rectangle. Goal: stop adjacent images from overlapping or being clipped on a printed card.

## Context
_Codebase facts and constraints learned during grilling._
- Packer (`src/lib/packer.ts`) exposes `packCircles(k, rng)` returning `PackedCircle[] {x, y, r}` in a normalised parent-unit frame (parent radius = 1). It is image-agnostic: takes only a count and a seeded RNG; radii derive from `PACKING_FRACTION = 0.55` and a ±25 % spread (`packer.ts:26, 55, 58–59`). No per-image size info is fed in today.
- The packer guarantees its returned circles do not overlap; today's visible overlap is a *renderer* issue, not a packer issue.
- Domain model is `UploadedImage = { id, file, url, name }` (inline in `src/App.tsx:34–39`). No `width`, `height`, `radius`, or pixel-geometry fields. `url` is a `URL.createObjectURL(file)` blob URL minted in `handleImagesAdded` (App.tsx:134–144).
- First decode of each image happens at generate-time via `loadImage` (App.tsx:50–56, called from `handleGenerate` App.tsx:181–183). There is no Canvas/`getImageData` pass anywhere — no alpha channel is ever read today.
- `drawCard` (`src/render/drawCard.ts:98–116`) draws each loaded image to fill the full `slotDiameter × slotDiameter` *bounding square* of the packed circle, regardless of the image's silhouette or transparent margin. This is the direct cause of the overlap: square fills square, but adjacent square corners poke beyond the inscribed circle into neighbouring slots.
- No persistence layer (no IndexedDB / localStorage). State is session-only React `useState`. Blob URLs are revoked on unmount.
- No image cache: every Generate click re-decodes all images via fresh `loadImage` calls. Inside one generate pass the same `HTMLImageElement` references are re-used across all 57 cards.
- Zero runtime image-pixel deps in `package.json`. `pdf-lib` is the only image-adjacent runtime dep (PNG embed only). Native browser `OffscreenCanvas` + `getImageData` is the zero-dep path. `jsdom` (test runtime) does not implement a real canvas (per `drawCard.test.ts:7` comment).

## Decisions
_Resolved through grilling. Each entry references the question that produced it._
1. **Scope — renderer-only.** Compute per-image silhouette geometry once per upload, then change `drawCard` so the image's silhouette circle maps exactly to the slot circle (instead of the image filling the slot's circumscribing square). The packer (`src/lib/packer.ts`) and its `PackedCircle` output are untouched. Rationale: this is the minimum-viable fix for the screenshot's overlap — the packer already returns non-overlapping circles; the bleed comes purely from drawing into circumscribing squares. Same per-image silhouette data feeds option 3 (NFP polygon packing) if we ever escalate, so this slice is also a stepping stone.
2. **Silhouette circle algorithm — Welzl's smallest enclosing circle on opaque-mask contour pixels.** Decode the image once, scan its alpha channel once to extract contour pixels (edge of opaque region via 4-neighbour check), then run Welzl's randomised SEC algorithm on that point set. Yields the geometrically optimal `{cx, cy, r}` in image-coordinate space. Rationale: the whole feature exists to stop wasting slot space on transparent padding — a centroid+max-radius approximation hands a noticeable chunk of that waste back. Welzl's is a well-known ~50–80 line implementation that unit-tests cleanly as a pure function on point arrays (no canvas, no jsdom problems).
3. **When — upload time, cached on the `UploadedImage` record.** `handleImagesAdded` runs the silhouette compute once per file as part of accepting the upload; the result lives on the `UploadedImage` record alongside `id`, `file`, `url`, `name`. Generate-time reads the cached value with zero extra work. Rationale: Generate is already the slowest user-facing path (decode + 57 cards × `drawCard` + buildPdf); piling per-image getImageData + Welzl on top would meaningfully degrade perceived latency, while at upload time the user is idle and the per-image cost is ~5–20 ms. Carrying the value on the record avoids a separate cache-invalidation surface when images are removed. Tradeoff accepted: each image is decoded twice in a session (once at upload for silhouette extraction, once at generate for rendering).
4. **Implementation seam — pure helper + thin canvas adapter.** Two modules: `src/lib/silhouette.ts` exports `computeSilhouetteCircle(alpha: Uint8ClampedArray, width: number, height: number, threshold: number) → { cx, cy, r }` (pure: alpha-mask scan → contour extraction → Welzl's SEC) and `src/lib/imageAlpha.ts` exports `extractAlphaMask(file: File | Blob) → Promise<{ width, height, alpha }>` (canvas-touching adapter). The pure layer is fully Vitest-covered with hand-crafted fixture arrays (cross, off-center disc, L-shape). The adapter is canvas-touching glue tested only via Playwright (jsdom has no canvas). Mirrors the existing `packer.ts` (pure) / `drawCard.ts` (canvas) split.
5. **Alpha threshold — `alpha > 0` (any non-zero alpha counts as silhouette).** Conservative: includes the full anti-aliased edge, so the silhouette circle never clips visible content. The cost is ~1–2 px of image-space slack at the AA halo, which is dwarfed by the slot-space win from killing the circumscribing-square draw. Rejected `alpha >= 128` (clips the AA halo, can read as a faint outline on high-DPI print) and `alpha >= 16` (middle ground but still trades correctness for a marginal tightening the user wouldn't see). No per-image UI knob — kept implicit.
6. **Pixel-reading API — `createImageBitmap(blob)` + `OffscreenCanvas`.** Modern async API: one call returns a fully-decoded `ImageBitmap` directly from the `File` blob, drawn once to an `OffscreenCanvas` at natural size, `getImageData` returns RGBA → keep the alpha channel only. No HTMLImageElement, no extra `URL.createObjectURL` minting, no `onload`/`onerror` race; errors surface as promise rejections. The existing `loadImage` helper (App.tsx:50–56, `new Image()` + onload) stays untouched and continues serving the generate-time render path — different lifecycle, different concern. Browser support is universal in evergreen targets; jsdom doesn't implement it but Decision 4's two-layer split already routes adapter coverage through Playwright.
7. **Failure modes at upload time.**
   - **Decode failure** (`createImageBitmap` rejects, corrupt/unsupported file): reject the upload, surface an amber notice via the existing notices list (matches the file-type-gate behaviour in `UploadDropzone`).
   - **Fully transparent image** (zero opaque pixels under the threshold): reject the upload, surface a specific notice ("this image has no visible content — nothing would print"). Today such files would silently render as blank; explicit rejection is friendlier.
   - **Huge image**: `extractAlphaMask` downsamples internally to max 512 px on the longest edge before `getImageData`. Welzl only needs contour positions, not pixel-perfect accuracy; the silhouette geometry is preserved within 1 image-space pixel. Render pipeline keeps the full-res original; the cap is private to the silhouette pass. Also caps upload-time CPU at ≤ 256 K pixels scanned per image, irrespective of source resolution.
8. **`UploadedImage` shape — required, normalised silhouette field.** Extend the inline type at `src/App.tsx:34–39` to `{ id, file, url, name, silhouette: { cx, cy, r } }`. All three numbers are normalised to the natural-image dimensions: `cx = pixelCx / naturalWidth`, `cy = pixelCy / naturalHeight`, `r = pixelR / naturalWidth` — all in [0, 1]. *Required*, not optional: Decision 7 routes every silhouette-extraction failure to upload rejection, so any record that exists in state carries a valid silhouette by construction (no `drawCard` fallback path needed). *Normalised*, not pixel: the silhouette pass downsamples internally (Decision 7), so storing raw pixel coords would leak the internal scan resolution into the type. Normalised values are stable across whatever internal scan size we choose later.
9. **Upload UX — sync-then-add (await `Promise.allSettled`, then commit).** `handleImagesAdded` becomes async: fire `extractAlphaMask` in parallel for every dropped file, await `Promise.allSettled`, then append the successes to `images` state in one `setState` call and post one notice per rejection. Bulk-drop wait: ≈ 5–20 ms × max-in-parallel; imperceptible to mildly perceptible. Optimistic-then-update was rejected because it requires a "partially-formed `UploadedImage`" placeholder type that conflicts with Decision 8's required-silhouette invariant.
10. **`drawCard` math + `PACKING_FRACTION` bump to 0.65.** `drawCard` replaces the slot-square fill with the silhouette mapping: `scale = slotR / (silhouette.r × img.naturalWidth)`, `drawX = slotCx − silhouette.cx × img.naturalWidth × scale`, `drawY = slotCy − silhouette.cy × img.naturalHeight × scale`, `drawW = img.naturalWidth × scale`, `drawH = img.naturalHeight × scale`. No fallback path (Decision 8 makes silhouette required). Separately, `PACKING_FRACTION` in `src/lib/packer.ts:26` bumps from `0.55` → `0.65` to recover the slot-edge space the circumscribing-square draw used to cover. **Correction (2026-05-15)**: the original claim that "0.65 is the safe ceiling" was wrong. The existing packer test only exercised a single fixed seed per k, so the silent over-pack went undetected. Production deck orders are k ∈ {3, 4, 6, 8, 12, 14} (derived from `SUPPORTED_PRIMES = [2, 3, 5, 7, 11, 13]` in `src/lib/orderPicker.ts` via k = n+1 per card). Empirical probe at fraction=0.65 over 200 seeds at k ∈ {6, 8, 13} (k=13 is an above-production stress value): ~12 % of seeds at k=8 and ~10 % at k=13 leave residual pair overlap after 600 iterations (worst case ~15 % of parent radius); k=6 failed at a similar rate. A separate sanity probe at k=32 and k=58 — well above any real deck order — converged cleanly, confirming the failure mode is concentrated in low-k regimes. Iteration cap is not the limit — bumping to 5000 barely moves the needle; the relaxation gets stuck in local minima. Decision 13 below handles this with convergence detection + retry-with-fresh-seed, which makes 0.65 reliable in practice.
11. **Slicing — four slices, fold the canvas adapter into the wiring slice.** Slices: (1) pure `silhouette.ts` SEC helper, (2) `imageAlpha.ts` adapter + `UploadedImage` type evolution + async `handleImagesAdded`, (3) `drawCard` silhouette consumption + `PACKING_FRACTION` bump, (4) packer convergence detection + retry-with-fresh-seed. Rationale: the canvas adapter is thin glue and untestable in isolation under jsdom (Decision 4); folding it into the wiring slice means it lands together with the App-level tests that mock the adapter at module boundary. Slice 4 was added 2026-05-15 after a live deck print surfaced the silent over-pack regression at fraction=0.65 — see Decision 10's correction. Each slice is one Red→Green→Refactor cycle.
12. **Card-back image — out of scope.** The card-back image uploaded via `PrintSettings` is drawn full-bleed across the back of every card by `buildPdf`; it never passes through `packCircles` or `drawCard`'s per-slot loop. Silhouette geometry would have no consumer. Deferred.
13. **Packer convergence detection + retry-with-fresh-seed (added 2026-05-15).** After the force-directed relaxation loop in `packCircles` finishes, scan the final layout for residual pair overlap above a small tolerance (`OVERLAP_TOLERANCE = 1e-4` in parent-unit space). If any pair overlaps, derive a new RNG seed from the caller-supplied `rng` (one `rng()` draw → uint32 → `mulberry32`) and re-run from scratch. Cap retries at `MAX_RETRIES = 8` per call; if all retries fail, throw a named `PackingDidNotConvergeError` carrying `{ k, attempts }`. Rationale: at fraction=0.65 the per-seed failure rate is ~12 % at k=8 and ~10 % at k=13 — geometric retry collapses that to ~10⁻⁸ across 8 attempts. App-level: `handleGenerate` catches the error and surfaces an amber notice (canonical copy: `"Could not pack card ${N} after ${attempts} attempts — please regenerate."`) rather than rendering a broken card. Determinism preserved: the retry seed chain is derived from the original seed, so a given `(k, rng)` pair still yields a single deterministic output. Per-card wall-clock impact for a 57-card deck (the n=7 production case, k=8 per card): expected ≈ 1.14× baseline; worst-case bounded by `MAX_RETRIES`. Retries are silent — no telemetry surface — but `attempts` is carried on the terminal `PackingDidNotConvergeError` for any future diagnostics.

## Slices

### Slice 1 — Pure silhouette helper (`src/lib/silhouette.ts`)
**Outcome:** A new pure module exports `computeSilhouetteCircle(alpha, width, height, threshold = 1) → { cx, cy, r }` returning the geometrically optimal smallest enclosing circle of the opaque-pixel set, in [0, 1]-normalised image-space (cx/cy normalised to width/height respectively; r normalised to width). Also exports a named `EmptySilhouetteError` for the fully-transparent case.
**Test (Red):** New file `src/lib/silhouette.test.ts` with five cases:
- "computes correct circle for a 7×7 centred cross" — `cx ≈ 0.5, cy ≈ 0.5, r` within tolerance.
- "computes correct circle for an off-centre solid disc" — 16×16 mask with disc at (4, 8), radius 3 → `cx ≈ 4/16, cy ≈ 8/16, r ≈ 3/16` (small tolerance for Welzl's randomised order).
- "computes correct circle for an L-shape" — Welzl finds the diagonal-spanning circle, smaller than the AABB-circumscribed circle.
- "throws `EmptySilhouetteError` when no pixels meet the threshold" — empty/all-transparent alpha mask.
- "handles a single opaque pixel as a degenerate circle" — `r = 0` (or `r = 0.5/dim` if the half-pixel convention is chosen during implementation).
**Implementation (Green):** Welzl's randomised SEC + 4-neighbour contour extraction over the alpha array. Threshold defaults to 1 (Decision 5). Pure: no canvas, no DOM, no globals. ~80–100 lines TS.
**Refactor:** None expected. If Welzl's recursion depth becomes a concern under tests with thousands of contour points, convert to iterative form (literature exists). Probably unnecessary at our scales.
**Acceptance:** All five tests pass; `npm run lint`, `npx tsc --noEmit` clean.

**Implementation (Green):**

- `src/lib/silhouette.ts` (UPDATE) — export `EmptySilhouetteError` for caller-side handling.

**Acceptance:** All three new tests pass; existing `App.test.tsx` tests still pass; `npm run lint`, `npx tsc --noEmit` clean; Playwright happy-path still typechecks. The Playwright run picks up the real adapter end-to-end when the user invokes it.

### Slice 3 — `drawCard` silhouette consumption + `PACKING_FRACTION` 0.55 → 0.65 bump
**Outcome:** Each rendered image's silhouette circle maps exactly to its slot circle (Decision 10 math). Slot density bumps from 0.55 → 0.65 to compensate for the recovered slot-edge space. End-to-end: the user's screenshot scenario (stopwatch + ball + shirt) no longer overlaps, and visible density remains close to the pre-feature feel.
**Test (Red):** Extend `src/render/drawCard.test.ts`:

- Update any existing drawCard test asserting the old slot-square draw shape so it matches the new call signature.
- If `src/lib/packer.test.ts` asserts a specific `baseRadius` numeric tied to `PACKING_FRACTION = 0.55`, bump the expected value to `sqrt(0.65 / k)`.
**Implementation (Green):**
- `src/render/drawCard.ts` (UPDATE) — replace the slot-square fill block (drawCard.ts:98–116) with the silhouette mapping math from Decision 10. No fallback path.
- `src/lib/packer.ts` (UPDATE) — set `PACKING_FRACTION = 0.65`.
- `src/render/drawCard.test.ts` (UPDATE).
- `src/lib/packer.test.ts` (UPDATE only if its asserts are value-tied to 0.55).
**Refactor:** None expected.
**Acceptance:** Updated + new drawCard test passes; packer tests pass at the new fraction; `npm run lint`, `npx tsc --noEmit` clean. The Playwright happy-path continues to pass (its asserts are semantic, not pixel-diff). The user can visually verify "no overlap" on a real generated card; this is a manual check, not automated.

### Slice 4 — Packer convergence detection + retry-with-fresh-seed (added 2026-05-15)
**Outcome:** `packCircles(k, rng)` is guaranteed to return a non-overlapping layout at `PACKING_FRACTION = 0.65` for all production deck orders (k ∈ {3, 4, 6, 8, 12, 14} — derived from `SUPPORTED_PRIMES = [2, 3, 5, 7, 11, 13]` in `src/lib/orderPicker.ts` via k = n+1 per card). Non-convergent seeds are silently retried with a derived fresh seed (up to `MAX_RETRIES = 8`); if every retry fails the function throws a named `PackingDidNotConvergeError` and `handleGenerate` surfaces an amber notice. Determinism preserved — a given `(k, rng)` pair still yields one deterministic output, because the retry seed chain is derived from the caller's `rng`.
**Test (Red):** Extend `src/lib/packer.test.ts`:
- "produces non-overlapping output across 200 seeds for k=8 and k=13 at PACKING_FRACTION=0.65" — replace the current single-seed `it.each([8, 13])` test with a 200-seed sweep that asserts `worstPair < OVERLAP_TOLERANCE` on every result. (This is the test that *should* have caught the regression in Slice 3. Note: k=13 here is a stress-test value above any real deck order — production k ∈ {3, 4, 6, 8, 12, 14} per `SUPPORTED_PRIMES` — kept for continuity with the existing test and as a margin probe.)
- "exports `PackingDidNotConvergeError` and throws it when every retry fails" — drive the packer with a stub RNG plus a documented test-only seam on `packer.ts` that forces every `attemptPacking` to return an overlapping layout; assert the thrown error carries `{ k, attempts: MAX_RETRIES }`. A purely pathological RNG isn't sufficient because the retry seed derivation immediately re-randomises via `mulberry32`, so almost any caller RNG eventually finds a converging chain.
- "produces identical output for the same caller-supplied rng" — call `packCircles(8, mulberry32(42))` twice; assert deep equality. Covers determinism of the public surface; does not necessarily exercise the retry chain (seed 42 may converge on attempt 1).
Extend `src/App.test.tsx`:
- "surfaces an amber notice when packCircles throws PackingDidNotConvergeError" — mock `packCircles` (or arrange via a stub RNG) to throw; click Generate; assert the notice appears and `renderedCards` is cleared per the existing `handleGenerate` failure path.
**Implementation (Green):**
- `src/lib/packer.ts` (UPDATE) — extract the existing seed → relax → clamp pipeline into a private `attemptPacking(k, rng)`. New top-level `packCircles` loops up to `MAX_RETRIES = 8` attempts: after each `attemptPacking`, scan for pair overlap > `OVERLAP_TOLERANCE = 1e-4`; on success return; on failure derive a fresh seed from the caller's `rng` (`mulberry32(Math.floor(rng() * 0xffffffff))`) and retry. After all retries, throw `PackingDidNotConvergeError`. Export the named error class.
- `src/App.tsx` (UPDATE) — wrap the per-card `packCircles` call in `handleGenerate` so a `PackingDidNotConvergeError` is caught, surfaced via the notices list with the canonical copy `"Could not pack card ${N} after ${attempts} attempts — please regenerate."`, and `renderedCards` is cleared (matches the existing error contract).
- `src/lib/packer.test.ts` (UPDATE), `src/App.test.tsx` (UPDATE).
**Refactor:** If `attemptPacking` extraction nudges the file past ~200 lines, split `seedCircles` / `relaxOnce` into a `relaxation.ts` peer. Probably unnecessary.
**Acceptance:** All new and existing tests pass; `npm run lint`, `npx tsc --noEmit` clean. The 200-seed sweep makes future density tuning trustable. The user can visually verify "no overlap" on a real generated 57-card deck — manual check, not automated.
